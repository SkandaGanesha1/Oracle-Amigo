Direct diagnosis

The right architecture is already partially present in your repo:

apps/voice-launcher/     → QuickVoice desktop mini app
src/voice/               → backend voice command layer
src/server.ts            → wires VoiceCommandService + registerVoiceCommandRoutes
local agent server       → should remain the brain
control plane relay      → routes requests to remote agents
remote local agent       → searches files and asks approval

The README says Quick Voice already exists as a Tauri v2 + React mini-launcher, opens with Ctrl+Space, defaults to http://127.0.0.1:3399, supports typed command previews, microphone permission/waveform UI, and exposes voice APIs such as GET /voice/status, POST /voice/commands, GET /voice/commands/:id, POST /voice/commands/:id/confirm, and POST /voice/commands/:id/cancel. It also says remote file commands create a preview first, then confirmation submits the relay file request.

But the current backend voice flow appears to be rule-parser-first, not LLM-parser-first. In the accessible VoiceCommandService.ts, createCommand() calls:

parseVoiceCommand(request.transcript)

and validates the result. I do not see an LLM parser being called in that path. That explains your symptom: the transcript may reach /voice/commands, but it is not being sent to an LLM for structured extraction.

The server wiring shows VoiceCommandService is created in src/server.ts, with resolveUser() using the cloud directory and executeRemoteFileRequest() using PeerRoutingService + RelayClient. That is good: the voice command backend is already in the correct place, inside the local agent. But the parser/execution lifecycle still needs hardening.

1. Correct integration model

Do not make oracle-amigo-quick-voice.exe own the workflow.

The correct model is:

oracle-amigo-quick-voice.exe
    ↓
transcript
    ↓
POST http://127.0.0.1:<local-agent-port>/voice/commands
    ↓
Oracle Amigo local agent server
    ↓
LLM/rule parser creates structured JSON
    ↓
local agent validates requester/target/action
    ↓
user confirms command
    ↓
local agent sends A2A relay request
    ↓
receiver agent searches files
    ↓
receiver approves
    ↓
file transfers back
    ↓
requester sees mission/receipt in Agentic Chat UI

So:

QuickVoice = input surface
Local Agent = brain / policy / workflow owner
Control Plane = relay / registry / presence
Remote Agent = executor on receiver device
Agentic Chat UI = mission timeline and user-visible state

This also aligns with A2A’s task/message model: A2A defines agent operations around messages, tasks, task state, artifacts, push notifications, and authenticated extended Agent Cards. Your QuickVoice command should become an Oracle Amigo mission and an A2A-style file request task, not a direct call from the launcher to the remote laptop.

3. Why it is not working now

Based on accessible code and your symptoms, these are the most likely root causes.

Root cause 1: transcript is not sent to an LLM

VoiceCommandService.createCommand() uses:

parseVoiceCommand(request.transcript)

directly. That means the current flow is deterministic parser only. If your expectation is “send transcript to LLM and get JSON,” that path is missing or not wired.

Fix

Introduce a parser interface:

export interface VoiceCommandParser {
  parse(input: VoiceCommandParserInput): Promise<VoiceCommandParseResult>;
}

Implement:

RuleBasedVoiceCommandParser
LlmVoiceCommandParser
HybridVoiceCommandParser

The hybrid parser should do:

1. Try rule parser.
2. If confidence >= 0.85, use it.
3. Otherwise call LLM parser.
4. Validate LLM JSON with Zod.
5. If invalid, repair once.
6. If still invalid, return clarification_required.

OpenAI Structured Outputs are relevant if you use OpenAI as the LLM provider, because they allow model outputs to follow a JSON schema rather than free-form text.

Root cause 2: confirmCommand() marks submitted commands as completed

In the accessible VoiceCommandService.ts, after executeRemoteFileRequest() returns, the code sets:

completedAt: result.status === "failed" ? null : new Date().toISOString()

That means if the result is "submitted", the command still gets a completedAt. This is wrong. A remote file request is not completed when it is submitted to relay. It is completed only after the receiver approves, transfer finishes, requester downloads, and hash verification succeeds.

Fix

Use:

completedAt:
  result.status === "completed"
    ? new Date().toISOString()
    : null

Better: represent the command as a mission and update it asynchronously.

submitted
→ waiting_remote_agent
→ waiting_receiver_approval
→ transferring
→ completed
Root cause 3: capability mismatch may stop target resolution

In src/server.ts, the voice remote file request path calls:

peerRouting.resolveTarget({
  peerUserId: targetUserId,
  capability: "file.request",
  cloud
})

But earlier architecture and capability names use values like:

file.request.search
file.transfer.receive
human.approval.request

If PeerRoutingService expects exact capability matching, then "file.request" may fail even when Docin’s agent supports "file.request.search".

Fix

Standardize capabilities:

const Capability = {
  FileRequestSearch: "file.request.search",
  FileTransferReceive: "file.transfer.receive",
  HumanApprovalRequest: "human.approval.request"
} as const;

Then in voice command execution:

capability: Capability.FileRequestSearch

Or support aliases:

file.request → file.request.search

but do not leave this ambiguous.

Root cause 4: QuickVoice may only create a preview, not confirm

The README says voice commands “create a preview first” and confirmation submits the relay file request.

So if the .exe only calls:

POST /voice/commands

but never calls:

POST /voice/commands/:id/confirm

then the backend will parse/preview but nothing will execute.

Fix

QuickVoice must do this sequence:

POST /voice/commands
→ show preview
→ user confirms
→ POST /voice/commands/:id/confirm
→ show submitted/waiting status

For high-confidence low-risk commands, you can add a user setting:

Auto-confirm remote file requests: off by default

Keep it off initially.

Root cause 5: QuickVoice request payload may not match backend schema

From VoiceCommandService, the request object seems to expect fields like:

request.transcript
request.source
request.locale
request.sttConfidence

If QuickVoice sends a nested object like:

{
  "transcript": "...",
  "stt": {
    "confidence": 0.91
  }
}

but your backend schema expects:

{
  "transcript": "...",
  "sttConfidence": 0.91
}

then the route may reject or silently ignore confidence.

Fix

Standardize the request schema:

export const VoiceCommandRequestSchema = z.object({
  transcript: z.string().trim().min(1).max(4000),
  source: z.enum(["quickvoice", "agentic_chat", "test"]).default("quickvoice"),
  locale: z.string().optional(),
  inputMode: z.enum(["speech", "typed"]).default("speech"),
  stt: z.object({
    provider: z.string().optional(),
    confidence: z.number().min(0).max(1).optional()
  }).optional(),
  sttConfidence: z.number().min(0).max(1).optional()
});

Normalize internally:

const sttConfidence =
  request.sttConfidence ?? request.stt?.confidence ?? null;
Root cause 6: STT may not actually be implemented

The README says QuickVoice supports microphone permission/waveform UI and typed command previews. That does not necessarily mean speech-to-text is complete.

If the UI captures audio and animates the waveform but never produces a transcript, then /voice/commands will never receive anything.

Fix

Implement STT behind an interface:

interface SpeechTranscriber {
  start(): Promise<void>;
  stop(): Promise<TranscriptResult>;
  onPartial(callback: (partial: string) => void): void;
}

Implement in phases:

MockTranscriber
BrowserSpeechTranscriber
WhisperSidecarTranscriber
CloudSttTranscriber later

For privacy-sensitive local desktop, whisper.cpp is a strong future sidecar option because it is a local C/C++ implementation of Whisper and can run on desktop without sending raw audio to the cloud.
